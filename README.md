## Архитектура

Проект написан по Clean Architecture — слои core, application, infrastructure, interfaces. Клиентская часть намеренно минимальна, упор на бэкенд.

```
core/            - доменные модели и абстрактные репозитории
application/     - use cases, dto, mappers
infrastructure/  - база данных, redis, jwt
interfaces/      - fastapi роуты и зависимости
```

Два сервиса: авторизация и контент. Авторизация выдаёт токены и управляет сессиями через Redis. Контент проверяет токен и роль через зависимости FastAPI.

## API

### POST /dummylogin

Быстрый вход без пароля, для демонстрации. Принимает role: "admin" или "user" и device_id.

```json
{ "role": "admin", "device_id": "my-device" }
```

Возвращает пару токенов.

### POST /register

```json
{ "email": "user@example.com", "password": "secret", "role": "user" }
```

Возвращает созданного пользователя, статус 201.

### POST /login

```json
{ "email": "user@example.com", "password": "secret", "device_id": "my-device" }
```

Возвращает пару токенов. Если уже была активная сессия на этом устройстве — старый access-токен сразу попадает в чёрный список.

### POST /logout

Требует заголовок `Authorization: Bearer <access_token>` и query-параметр `device_id`. Кладёт access-токен в чёрный список, удаляет обе записи из белого.

### POST /refresh

Требует заголовок `Authorization: Bearer <refresh_token>` и query-параметр `device_id`. Возвращает новую пару токенов, старый refresh аннулируется.

### GET /content/...

Все контентные роуты требуют валидный access-токен в заголовке.

| Роут | Кто имеет доступ |
|------|-----------------|
| /content/all | все, без авторизации |
| /content/common | admin и user |
| /content/admin | только admin |
| /content/user | только user |

## Как работают белый и чёрный список

### Пара токенов

Вместо одного токена система выдаёт два:

- access-токен — короткоживущий (15 минут), используется в каждом запросе
- refresh-токен — долгоживущий (7 дней), используется только для получения новой пары

Это стандартный подход: украденный access-токен протухает быстро, а refresh-токен защищён ротацией.

### Белый список

Ключи в Redis:

```
whitelist:access:{user_id}:{device_id}  - текущий access-токен, TTL 15 минут
whitelist:refresh:{user_id}:{device_id} - текущий refresh-токен, TTL 7 дней
```

На одно устройство хранится только один активный токен каждого типа. При каждом логине или рефреше:

1. Читаем старый access из белого списка
2. Если он есть — кладём в чёрный список с оставшимся TTL
3. Записываем новый access и новый refresh в белый список

### Чёрный список

Ключ: `blacklist:token:{raw_jwt}`, значение "1", TTL = остаток жизни токена.

Токен попадает в чёрный список в трёх случаях: при logout, при повторном логине с того же устройства, при рефреше (старый access отзывается автоматически).

### Проверка токена в каждом запросе

1. Декодируем токен, проверяем подпись и срок
2. Проверяем тип — должен быть "access"
3. Ищем в чёрном списке — если есть, возвращаем 401 "token revoked"
4. Если передан device_id — проверяем белый список. Если пусто — 401 "session expired". Если не совпадает — 401 "token superseded"

### Refresh token rotation

При рефреше:

1. Проверяем подпись и тип refresh-токена
2. Сверяем с тем, что хранится в Redis для этого устройства
3. Если токены не совпадают — кто-то пытается использовать уже заменённый токен. Сессия устройства полностью сбрасывается, возвращаем 401
4. Если всё ок — выдаём новую пару, старые ключи перезаписываются

## Роли и контент

Роль хранится прямо в JWT payload:

```json
{ "user_id": "uuid", "role": "admin", "type": "access", "exp": 1234567890 }
```

Проверка через фабрику зависимостей `require_role(*roles)`. Можно передать одну роль или несколько:

```python
@router.get("/common")
async def common(user=Depends(require_role("admin", "user"))):
    ...

@router.get("/admin")
async def admin(user=Depends(require_role("admin"))):
    ...
```

Реализовано две роли: admin и user. Есть общий контент для обеих и отдельный для каждой.

## Почему JWT + Redis

Рассматривал три варианта.

Чистый stateless JWT — просто, не нужен Redis, хорошо масштабируется. Но нельзя отозвать токен до истечения TTL. Украденный токен работает до expire — это неприемлемо.

Opaque tokens (сессии целиком в Redis) — полный контроль. Но каждый запрос идёт в Redis за данными пользователя, нет стандартного формата, всё самописное.

JWT + Redis whitelist/blacklist — выбрал этот. JWT самодостаточен (роль уже внутри, лишний запрос в БД не нужен), при этом токен можно отозвать в любой момент через чёрный список. Redis хранит только лёгкие строковые ключи, не полные данные сессии. Единственный минус — зависимость от Redis, но это решается репликацией.

## Безопасность и утечки токенов

### Как токен может утечь

- XSS — если токен в localStorage, вредоносный JS его прочитает
- MITM — перехват по незашифрованному каналу
- Утечка логов — токен попал в логи сервера в открытом виде
- Компрометация клиента — вирус, keylogger

### Как обнаружить кражу

В проекте реализован refresh token rotation с детектированием повторного использования. Если атакующий получил refresh-токен и использует его, пока легитимный пользователь тоже пытается сделать рефреш — один из них получит "token reuse detected" и вся сессия устройства будет сброшена. Ни один из них не продолжит работу со старым токеном.

Дополнительно: один пользователь + одно устройство = один активный токен. При повторном логине старый токен немедленно отзывается через чёрный список, а не просто вытесняется.

Что ещё можно добавить (в проекте не реализовано):

- Fingerprinting — сохранять User-Agent и IP при выдаче токена, сравнивать при каждом запросе
- Geo-velocity detection — логин из одной страны, запрос через минуту из другой

### Меры защиты

| Угроза | Что делать |
|--------|-----------|
| XSS | httpOnly cookie вместо localStorage |
| MITM | только HTTPS, HSTS |
| Долгоживущий токен | короткий TTL access + refresh rotation (реализовано) |
| Параллельные сессии | single-session whitelist per device (реализовано) |
| Украденный access | blacklist при logout и при повторном логине (реализовано) |
| Украденный refresh | refresh token rotation + reuse detection (реализовано) |
| Утечка секрета | JWT_SECRET_KEY из env, не хардкодить |

```
    core/
        domain/models.py          - доменные модели (dataclass)

        mappers/

    infrastructure/
        auth/jwt.py               - создание и декодирование токенов

```
